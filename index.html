Autonomous Motion

Created an autonomous robot using multiple ultrasonic sensors and programmed the robot to avoid objects which it encounters and choose the optimal path

Overview

As a project to complete in my free time I created an autonomous robot using multiple ultrasonic sensors in combination with an Arduino Microcontroller and a robot vehicle chassis. I was able to experiment with a variety of concepts such as Analog vs. Digital, Sensor Interfacing, Motor Control, and many more. My troubleshooting skills were also heavily enhanced as there were a variety of problems I was forced to overcome through perseverance when testing the robot for functionality.

Sensors

In order to provide the robot with a solid sense of awareness for its surroundings, I equipped it with 3 ultrasonic sensors. One sensor was placed at the front, whereas the other two were placed facing right and left. This allowed the robot to have a sense of whether there were objects directly ahead of it, as well as if there were any objects on its side. The sensors simply returned a number representing how long it took for the ultrasonic wave to bounce off the nearest object and return to the sensor. I then used this number along with a combination of the calculated speed of sound (according to the specified temperature) to calculate the distance to the nearest object.

Path Planning

The path planning for this robot was a simple control flow algorithm to determine the robots' actions dependant on its surroudings. The data received from each of the sensors was processed and operated on to determine the distance from the front of the vehicle to the nearest object, and then used to essentially stitch together a basic understanding of the robots' current surroundings. Dependant on the proximity of the nearest object at the front, or the nearest object at the left or right side, the robot would control its motors such that it moves away from any obstructions

Line Following Robot

Created a line following robot using 4 QTI infrared sensors and a motor controller to control motors.

Created a base for the robot using plexiglass and wheels, serving as the body of the robot, and attached wheels as well as a breadboard and adruino board to act as the control center for the robot. Attached two infrared sensors at the front of the robot in order to intake information about the ground over which the robot is travelling. Controlled the motors to the arduino board using an L293D chip. If both sensors read high, then the robot travelled forward along the line. If the robot veered to the side and was not completely on the line, then one sensor would read high whereas the other would read low, causing the motors to act accordingly and readjust so that the line is between the sensors once again.

Binary Counter

A breadboarding project consisting of multiple chips as well as 8 segment displays, wired to count from 0-99.

As a project to complete in my free time I decided to create a binary counter. The counter consists of two 7 segment display chips, an NE555 chip, an 74LS90 chip, and two CD4056BE chips. The NE555 chip acted as the interval setter between number changes as its resistances could be changed to increase or decrease the time between numbers. The 74LS90 chip is a decade counter, which had the job of counting from 0-9 in binary cyclically, and the binary counted was displayed by lighting up the appropriate LEDs. The CD4056BE chip acted as a digital display driver, as it intook the binary signals from the 74LS90 chip and converted them into digital signals, which were then relayed to the seven segment displays in order to display the numbers.

DigiCeipt

Developed a full-stack web application using the MEAN stack.

A screenshot of the product currently in its medium fidelity phase (not fully functional)

An early stage low level prototype for the home-page

Purpose

DigiCeipt was created as a method for individuals to be able to store their receipts digitally, rather than having to cope with the hassle of storing physical receipts. It began when I came home from university and saw my brothers storing some scattered physical receipts, which led me to decide that I wanted to create a digital storage solution to this problem.

Design

Problem Space

I began by understanding the problem space, potential users, and drafting a Situation Impact Statement to place some perspective on the problem I was tackling as well as the intended outcome. Then I moved forward to draft some personas using both primary information and secondary information.

Potential Solutions

Once I had a solid understanding of the problem space, I proceeded to break the problem into small functionalities that would be used to solve it. Once I'd identified these functionalities, I used a Morphological Chart to brainstorm a variety of potential solutions.

Solution Evaluation

Once my potential solutions were identified, I began to use a Quality Function Deployment (QFD) chart to undergo an in depth analysis and comparison of each of the options. By the end of the QFD, I had a certain solution scoring better than the other that I decided to proceed forward with.

Prototyping

With my solution now in mind, I started with a low-fidelity prototype of the solution. I didn't begin with a wireframe, but rather with a diagram showing the architecture of the application and the connections between each of the components. Once I had the functionality of the product planned out, I moved onto some UI low-level prototypes by just sketching out some interfaces. I'm currently in the medium-fidelity stage as my product isn't quite ready for production, but it is approaching a level of full functionality.

Iterate

This phase will continue forever!

Technical Details

Frontend

The dynamic client-side was developed using the Angular Framework and TypeScript. HTTP requests were sent to the server using the HTTP module and request results were subscribed to using Observables. Routing was incorporated to allow for multiple different views in a Single Page Application. Angular Material was used to contribute to a smooth and enjoyable user interface for users

Backend

A REST API was developed using an Express backend and Mongoose, implementing HTTP request methods to allow communication between the client-side and database. HTTP methods such as GET and PUT, and DELETE were defined at different endpoints in order to allow users to upload, edit, and delete receipts. API testing was completed using Postman, with the use of both manual and automated tests to ensure the endpoints were working according to expectations. MongoDB was used as the database of choice, the main reason being its scalability and ability to adapt to different storage formats as the need for/type of storage evolves.

Zombie Apocalypse Game

Developed a HUD zombie apocalypse game using C++

Purpose

Overall, Zombie Apoc was a project for me to explore the world of Object-Oriented-Programming and learn some different aspects of it to become comfortable with the concepts, as well as learn some good development practices.

Classes

These concepts ranged from a wide diversity, including classes to represent game components, such as a class for the player, as well as many more classes for different components such as zombies and accessories. Singleton classes were also incorporated to represent certain objects of which only one instance could exist, as well as static class implementations as well. Inheritance was also incorporated to allow child classes to acquire properties and methods from a base class.

Pointer Implementation

Pointers were implemented for memory maintenance and dynamic memory allocation. Memory was allocated when necessary for use and deleted when no longer necessary, ensuring optimal use of memory. I was able to learn about good memory allocation practices through this project and was also educated on dangling pointers, garbage collection, and programming to ensure that the memory heap is being used optimally.

Libraries

The SFML Library was used for the game's development and its many classes as well as class methods were used in the development process for a variety of areas, such as graphics, audio, and many more areas. This was an introduction for me to using libraries in C++ as I'd never used external C++ libraries before. It allowed me to gain comfort using libraries and their features, which has made me more comfortable as I continue to explore libraries such as STL and many more.



OutWeather

Developed a full-stack application using Python's flask framework as an entry to the University of Toronto's hackathon, Hack The Valley

description

OutWeather is a web application that helps users determine what to wear and what accessories to bring with them when they go outside.

Purpose

This application hopes to to advise users of the weather conditions and suggest an ideal outfit so that they are prepared for the weather conditions. A clean and simple UI was designed to make it easy to use. Users can enter their location, and the app will then proceed to display details on the weather, as well as a possible outfit to wear from top to bottom. A variety of display design principles were used in designing the display such as Norman's' Seven Stages of Action. These principles helped us design an interface to ensure easy interpretation for users, as well as easy execution of the action they would like to perform.

Technical Details

Python's Flask framework was used to develop the backend of this web application. User inputted data was sent to the OpenWeather API to retrieve weather information for the specified city. Jinja2 was then used to inject data into the HTML.
